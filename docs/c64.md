## Overview

The C64 Ultimate devices expose a REST API over HTTP on the device's LAN IP. The API enables full remote control: running programs, managing floppy drives, reading/writing C64 memory, streaming video/audio, and configuring every device setting.

**Supported devices:** Ultimate 64, Ultimate II+, Ultimate II+L
**Minimum firmware:** 3.11 (authentication added in 3.12)
**API version:** 0.1
**Transport:** HTTP (plain, no TLS)
**Base URL:** `http://<device-ip>/v1/`

---

## URL Format

```
/v1/<route>/<path>:<command>?<arguments>
```

| Component     | Description                                                        |
| ------------- | ------------------------------------------------------------------ |
| `/v1/`        | API version prefix (currently only `v1`)                           |
| `<route>`     | Resource group: `version`, `info`, `runners`, `configs`, `machine`, `drives`, `streams`, `files` |
| `<path>`      | Optional sub-resource (drive letter, stream name, file path, config category) |
| `:<command>`  | Action to perform on the resource                                  |
| `?<arguments>`| Query string parameters (key=value pairs)                          |

**Important:** Always use relative URLs (`/v1/...`). Paths containing spaces must be URL-encoded (`%20`).

---

## HTTP Verbs

| Verb   | Semantics                                                                                       |
| ------ | ----------------------------------------------------------------------------------------------- |
| `GET`  | Retrieves information without changing state                                                    |
| `PUT`  | Performs an action using information in the URL or referencing a file on the device's filesystem |
| `POST` | Performs an action using binary data attached to the request body                                |

The distinction between PUT and POST is consistent across the API: **PUT references files by path on the device**, while **POST uploads file data as a binary attachment**.

---

## Authentication

Since firmware **3.12**, if a "Network Password" is configured on the device, **all** API requests must include:

```
X-Password: <your-password>
```

| Scenario                  | Result                        |
| ------------------------- | ----------------------------- |
| No password configured    | Header not required           |
| Correct password provided | Request proceeds normally     |
| Missing or wrong password | `403 Forbidden` HTTP response |

## Response Format

### JSON Responses (`application/json`)

Most endpoints return JSON. **Every** JSON response includes an `errors` array:

```json
{
  "errors": []
}
```

**Critical:** A `200 OK` status does **not** guarantee success. The client must always check the `errors` array:

| `errors` Array | Meaning                                                      |
| -------------- | ------------------------------------------------------------ |
| Empty `[]`     | Operation succeeded; additional data may be present in the response |
| Non-empty      | Application-level error; array contains string error messages |

### Binary Responses (`application/octet-stream`)

Some endpoints (e.g., `readmem`) return raw binary data on success. A `200 OK` with binary content type indicates success.

### Response Handling Logic

| HTTP Status | Content-Type             | `errors`  | Result              | Action                                    |
| ----------- | ------------------------ | --------- | ------------------- | ----------------------------------------- |
| 200         | `application/json`       | Empty     | **Success**         | Process response data                     |
| 200         | `application/json`       | Non-empty | **App Error**       | Display error messages from array         |
| 200         | `application/octet-stream` | N/A     | **Success (binary)**| Process raw binary data                   |
| 4xx / 5xx   | Any                      | N/A       | **HTTP Error**      | Parse body as JSON for errors; fall back to status text |

---

## Error Handling

- There is **no automatic retry** logic in the API
- On error, display the message and clear it when the next operation succeeds
- Non-200 HTTP responses **may still** contain a JSON body with an `errors` array — always attempt to parse it
- For binary response endpoints, errors may arrive as ArrayBuffer that needs decoding before JSON parsing

## Endpoints

### About

#### GET `/v1/version`

Returns the API version.

**Parameters:** None

**Response:**
```json
{
  "version": "0.1",
  "errors": []
}
```

Returns device information.

**Parameters:** None

**Response:**
```json
{
  "product": "Ultimate 64",
  "firmware_version": "3.12",
  "fpga_version": "11F",
  "core_version": "143",
  "hostname": "Terakura",

  "errors": []
}
```

| Field              | Type   | Notes                           |
| ------------------ | ------ | ------------------------------- |

| `firmware_version` | string | Firmware version                |
| `fpga_version`     | string | FPGA bitstream version (hex)    |
| `core_version`     | string | Core version (U64 only)         |
| `hostname`         | string | Network hostname                |
| `unique_id`        | string | Unique device identifier (hex)  |

### Runners

Endpoints for loading and executing files on the C64. PUT variants reference files on the device's filesystem; POST variants accept uploaded binary attachments.

#### PUT `/v1/runners:sidplay`

Play a SID file from the device's filesystem.

| Parameter | Required | Description                                    |
| --------- | -------- | ---------------------------------------------- |
| `file`    | Yes      | Path to SID file on device                     |
| `songnr`  | No       | Song number to play (defaults to file's default song) |

**Notes:** The player will attempt to find a corresponding song lengths file in the `SONGLENGTHS` subdirectory.

---

#### POST `/v1/runners:sidplay`

Play an uploaded SID file.

| Parameter | Required | Description                           |
| --------- | -------- | ------------------------------------- |
| `songnr`  | No       | Song number to play                   |

**Request body:** SID file as binary attachment. An optional second attachment can be a song lengths file.

---

#### PUT `/v1/runners:modplay`

Play an Amiga MOD file from the device's filesystem.

| Parameter | Required | Description                    |
| --------- | -------- | ------------------------------ |
| `file`    | Yes      | Path to MOD file on device     |

---

#### POST `/v1/runners:modplay`

Play an uploaded Amiga MOD file.

**Request body:** MOD file as binary attachment.

---

#### PUT `/v1/runners:load_prg`

Load a PRG file into memory without running it. The machine resets and loads the program via DMA.

| Parameter | Required | Description                    |
| --------- | -------- | ------------------------------ |
| `file`    | Yes      | Path to PRG file on device     |

---

#### POST `/v1/runners:load_prg`

Load an uploaded PRG file into memory without running it.

**Request body:** PRG file as binary attachment.

---

#### PUT `/v1/runners:run_prg`

Load and auto-run a PRG file from the device's filesystem. The machine resets, loads via DMA, then runs automatically.

| Parameter | Required | Description                    |
| --------- | -------- | ------------------------------ |
| `file`    | Yes      | Path to PRG file on device     |

---

#### POST `/v1/runners:run_prg`

Load and auto-run an uploaded PRG file.

**Request body:** PRG file as binary attachment.

---

#### PUT `/v1/runners:run_crt`

Start a cartridge from the device's filesystem. The machine resets with the cartridge active. Does not alter device configuration.

| Parameter | Required | Description                    |
| --------- | -------- | ------------------------------ |
| `file`    | Yes      | Path to CRT file on device     |

---

#### POST `/v1/runners:run_crt`

Start an uploaded cartridge file.

**Request body:** CRT file as binary attachment.

### Configuration

Full read/write access to all device configuration settings. Wildcard patterns (`*`) are supported in category and item names.

#### GET `/v1/configs`

List all configuration categories.

**Parameters:** None

**Response:**
```json
{
  "categories": [
    "Audio Mixer",
    "SID Sockets Configuration",
    "UltiSID Configuration",
    "SID Addressing",
    "C64 and Cartridge Settings",
    "U64 Specific Settings",
    "Clock Settings",
    "Network settings",
    "WiFi settings",
    "Modem Settings",
    "LED Strip Settings",
    "Data Streams",
    "Software IEC Settings",
    "User Interface Settings",
    "Tape Settings",
    "Drive A Settings",
    "Drive B Settings"
  ],
  "errors": []
}
```

---

#### GET `/v1/configs/<category>`

Get all configuration items in a category. Wildcards allowed.

**Example:** `GET /v1/configs/drive%20a*`

**Response:**
```json
{
  "Drive A Settings": {
    "Drive": "Enabled",
    "Drive Type": "1541",
    "Drive Bus ID": 8,
    "ROM for 1541 mode": "1541.rom",
    "ROM for 1571 mode": "1571.rom",
    "ROM for 1581 mode": "1581.rom",
    "Extra RAM": "Disabled",
    "Disk swap delay": 1,
    "Resets when C64 resets": "Yes",
    "Freezes in menu": "Yes",
    "GCR Save Align Tracks": "Yes",
    "Leave Menu on Mount": "Yes"
  },
  "errors": []
}
```

---

#### GET `/v1/configs/<category>/<item>`

Get detailed info for a specific configuration item including value constraints. Wildcards allowed in both category and item.

**Example:** `GET /v1/configs/drive%20a*/*bus*`

**Response:**
```json
{
  "Drive A Settings": {
    "Drive Bus ID": {
      "current": 8,
      "min": 8,
      "max": 11,
      "format": "%d",
      "default": 8
    }
  },
  "errors": []
}
```

| Field     | Type           | Description                                              |
| --------- | -------------- | -------------------------------------------------------- |
| `current` | string or int  | Current value                                            |
| `min`     | int            | Minimum allowed value (for numeric/enum items)           |
| `max`     | int            | Maximum allowed value (for numeric/enum items)           |
| `format`  | string         | C-style format string (`%d` for numeric, `%s` for enum) |
| `default` | string or int  | Factory default value                                    |

---

#### PUT `/v1/configs/<category>/<item>?value=<value>`

Set a single configuration item.

| Parameter | Required | Description                    |
| --------- | -------- | ------------------------------ |
| `value`   | Yes      | New value for the item         |

**Example:** `PUT /v1/configs/drive%20a*/*bus*?value=9`

---

#### POST `/v1/configs`

Bulk-set multiple configuration items in one request.

**Content-Type:** `application/json`

**Request body:**
```json
{
  "Drive A Settings": {
    "Drive": "Enabled",
    "Drive Type": "1581",
    "Drive Bus ID": 8
  },
  "Drive B Settings": {
    "Drive": "Disabled"
  }
}
```

The JSON structure mirrors the format returned by GET requests.

---

#### PUT `/v1/configs:load_from_flash`

Restore the complete configuration from non-volatile memory (flash).

---

#### PUT `/v1/configs:save_to_flash`

Save the current configuration to non-volatile memory. These settings persist across reboots.

---

#### PUT `/v1/configs:reset_to_default`

Reset all settings to factory defaults. **Does not** modify the values stored in flash — only the in-memory (active) config is reset.

---

### Machine

Direct control of the C64 machine state, including memory access via DMA.

#### PUT `/v1/machine:reset`

Send a reset signal. Current configuration is unchanged.

---

#### PUT `/v1/machine:reboot`

Restart the machine. Re-initializes cartridge configuration, then resets.

---

#### PUT `/v1/machine:pause`

Pause the CPU by pulling the DMA line low. **Timers continue running** — only the CPU is halted.

---

#### PUT `/v1/machine:resume`

Resume from paused state. Releases the DMA line and the CPU continues from where it stopped.

---

#### PUT `/v1/machine:poweroff`

Power off the machine. **U64 only.** You will likely not receive a valid HTTP response since the device powers down.

---

#### PUT `/v1/machine:menu_button`

Simulate pressing the Menu button (Ultimate cartridge) or Multi Button (U64). Toggles the Ultimate menu overlay.

---

#### PUT `/v1/machine:writemem`

Write up to **128 bytes** to C64 memory via DMA using hex-encoded data in the URL.

| Parameter | Required | Format      | Description                           |
| --------- | -------- | ----------- | ------------------------------------- |
| `address` | Yes      | Hexadecimal | Starting memory address (e.g., `D020`) |
| `data`    | Yes      | Hexadecimal | Byte string (e.g., `0504`)            |

**Example:** `PUT /v1/machine:writemem?address=D020&data=0504`
Writes `$05` to `$D020` (green border) and `$04` to `$D021` (purple background).

**Limitations:**
- Maximum 128 bytes per request
- Cannot write to 6510 I/O registers

---

#### POST `/v1/machine:writemem`

Write a larger block of binary data to C64 memory via DMA.

| Parameter | Required | Format      | Description            |
| --------- | -------- | ----------- | ---------------------- |
| `address` | Yes      | Hexadecimal | Starting memory address |

**Request body:** Raw binary data (`application/octet-stream`).

**Constraint:** Data must not wrap around the `$FFFF` address boundary.

---

#### GET `/v1/machine:readmem`

Read C64 memory via DMA. Returns raw binary data.

| Parameter | Required | Format      | Description                       |
| --------- | -------- | ----------- | --------------------------------- |
| `address` | Yes      | Hexadecimal | Starting memory address           |
| `length`  | No       | Decimal     | Number of bytes to read (default: 256) |

**Response:** Binary data (`application/octet-stream`).

---

#### GET `/v1/machine:debugreg`

Read the debug register at `$D7FF`. **U64 only.**

**Response:**
```json
{
  "value": "XX",
  "errors": []
}
```

The value is returned in hexadecimal format.

---

#### PUT `/v1/machine:debugreg`

Write to the debug register at `$D7FF`, then read it back. **U64 only.**

| Parameter | Required | Format      | Description              |
| --------- | -------- | ----------- | ------------------------ |
| `value`   | Yes      | Hexadecimal | Value to write to `$D7FF` |

**Response:**
```json
{
  "value": "XX",
  "errors": []
}
```

### Floppy Drives

Manage the emulated floppy drives on the IEC bus. Drive identifiers are `a` and `b`. A third entry, `softiec`, represents the Software IEC (DOS emulation) drive.

#### GET `/v1/drives`

Get information about all drives.

**Response:**
```json
{
  "drives": [
    {
      "a": {
        "enabled": true,
        "bus_id": 8,
        "type": "1581",
        "rom": "1581.rom",
        "image_file": "",
        "image_path": ""
      }
    },
    {
      "b": {
        "enabled": false,
        "bus_id": 9,
        "type": "1541",
        "rom": "1541.rom",
        "image_file": "",
        "image_path": ""
      }
    },
    {
      "softiec": {
        "enabled": false,
        "bus_id": 11,
        "type": "DOS emulation",
        "last_error": "73,U64IEC ULTIMATE DOS V1.1,00,00",
        "partitions": [
          { "id": 0, "path": "/Temp/" }
        ]
      }
    }
  ],
  "errors": []
}
```

| Field        | Type    | Notes                                      |
| ------------ | ------- | ------------------------------------------ |
| `enabled`    | boolean | Whether the drive is active on the bus     |
| `bus_id`     | int     | IEC bus address (8–11)                     |
| `type`       | string  | `"1541"`, `"1571"`, `"1581"`, or `"DOS emulation"` |
| `rom`        | string  | Currently loaded ROM filename              |
| `image_file` | string  | Mounted disk image filename (empty if none)|
| `image_path` | string  | Full path to mounted image (empty if none) |
| `last_error` | string  | DOS error string (softiec only)            |
| `partitions` | array   | Partition table (softiec only)             |

---

#### PUT `/v1/drives/<drive>:mount`

Mount a disk image from the device's filesystem.

| Parameter | Required | Description                                        |
| --------- | -------- | -------------------------------------------------- |
| `image`   | Yes      | Path to disk image on device                       |
| `type`    | No       | Image type: `d64`, `g64`, `d71`, `g71`, `d81`. Inferred from extension if omitted |
| `mode`    | No       | Mount mode (see below)                             |

**Mount modes:**

| Mode        | Write Protected | Saves to Image |
| ----------- | --------------- | -------------- |
| `readwrite` | No              | Yes            |
| `readonly`  | Yes             | No             |
| `unlinked`  | No              | No (changes are discarded) |

---

#### POST `/v1/drives/<drive>:mount`

Mount an uploaded disk image.

| Parameter | Required | Description                                        |
| --------- | -------- | -------------------------------------------------- |
| `type`    | No       | Image type. Inferred from uploaded filename if omitted |
| `mode`    | No       | Mount mode (see table above)                       |

**Request body:** Disk image as binary attachment.

---

#### PUT `/v1/drives/<drive>:reset`

Reset the specified drive.

---

#### PUT `/v1/drives/<drive>:remove`

Remove (unmount) the disk from the drive.

---

#### PUT `/v1/drives/<drive>:on`

Turn the drive on. If already on, performs a reset.

---

#### PUT `/v1/drives/<drive>:off`

Turn the drive off. It becomes inaccessible on the IEC bus.

---

#### PUT `/v1/drives/<drive>:load_rom`

Load a custom ROM into the drive from the device's filesystem. **Temporary** — reverts on reboot or drive mode change.

| Parameter | Required | Description                                     |
| --------- | -------- | ----------------------------------------------- |
| `file`    | Yes      | Path to ROM file (must be 16K or 32K depending on drive type) |

---

#### POST `/v1/drives/<drive>:load_rom`

Load a custom ROM from an uploaded file. Same constraints as PUT variant.

**Request body:** ROM file as binary attachment (16K or 32K).

---

#### PUT `/v1/drives/<drive>:set_mode`

Change the drive's emulation mode. **Also reloads the default ROM**, overwriting any temporary ROM.

| Parameter | Required | Values                |
| --------- | -------- | --------------------- |
| `mode`    | Yes      | `1541`, `1571`, `1581` |

### Data Streams (U64 Only)

The U64 can stream real-time video, audio, and debug data over UDP to a target IP.

#### PUT `/v1/streams/<stream>:start`

Start a data stream.

| Parameter  | Required | Description                                            |
| ---------- | -------- | ------------------------------------------------------ |
| `<stream>` | Yes      | Stream name: `video`, `audio`, or `debug` (URL path)  |
| `ip`       | Yes      | Target IP address. Optional port via `ip:port` syntax  |

**Default ports:**

| Stream  | Default Port |
| ------- | ------------ |
| `video` | 11000        |
| `audio` | 11001        |
| `debug` | 11002        |

**Example:** `PUT /v1/streams/audio:start?ip=192.168.1.100:6789`

**Note:** Starting the `video` stream automatically **stops** the `debug` stream.

---

#### PUT `/v1/streams/<stream>:stop`

Stop a running stream.

| Parameter  | Required | Description                                          |
| ---------- | -------- | ---------------------------------------------------- |
| `<stream>` | Yes      | Stream name: `video`, `audio`, or `debug` (URL path) |

---

### File Manipulation

Create disk image files on the device's filesystem. The file info endpoint is noted as **unfinished** in the official docs.

#### GET `/v1/files/<path>:info`

Get basic file information (size, extension). Supports wildcards. **Unfinished feature.**

---

#### PUT `/v1/files/<path>:create_d64`

Create a new `.d64` disk image.

| Parameter  | Required | Description                                          |
| ---------- | -------- | ---------------------------------------------------- |
| `<path>`   | Yes      | Full filesystem path including filename (URL path)   |
| `tracks`   | No       | Number of tracks: `35` (default) or `40`             |
| `diskname` | No       | Disk header name (defaults to filename)              |

---

#### PUT `/v1/files/<path>:create_d71`

Create a new `.d71` disk image. Fixed at **70 tracks**.

| Parameter  | Required | Description                                          |
| ---------- | -------- | ---------------------------------------------------- |
| `<path>`   | Yes      | Full filesystem path including filename (URL path)   |
| `diskname` | No       | Disk header name (defaults to filename)              |

---

#### PUT `/v1/files/<path>:create_d81`

Create a new `.d81` disk image. Fixed at **160 tracks** (80 per side).

| Parameter  | Required | Description                                          |
| ---------- | -------- | ---------------------------------------------------- |
| `<path>`   | Yes      | Full filesystem path including filename (URL path)   |
| `diskname` | No       | Disk header name (defaults to filename)              |

---

#### PUT `/v1/files/<path>:create_dnp`

Create a new `.dnp` disk image.

| Parameter  | Required | Description                                          |
| ---------- | -------- | ---------------------------------------------------- |
| `<path>`   | Yes      | Full filesystem path including filename (URL path)   |
| `tracks`   | Yes      | Number of tracks (1–255). Each track = 256 sectors. Max ~16 MB |
| `diskname` | No       | Disk header name (defaults to filename)              |
